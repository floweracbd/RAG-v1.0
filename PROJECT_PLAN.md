# PROJECT PLAN

## 1. 项目名称

面向电商客服场景的单 Agent RAG 知识服务系统

## 2. 项目目标

构建一个可以支撑电商客服场景知识问答的 AI 应用后端。系统支持文档上传、知识入库、向量检索、RAG 问答以及单 Agent 工具调用，具备清晰的数据流、模块划分和基本工程结构。

本项目的核心目标不是追求炫技，而是完成一个具备以下特点的项目：

1. 能跑通
2. 能讲清楚
3. 能继续扩展
4. 能支撑初级 AI 应用开发岗位面试

## 3. 项目范围

### 3.1 第一版必须实现

1. 文档上传与本地保存
2. 基于 MD5 的重复校验
3. 多格式文档解析
4. 文本切块
5. embedding 生成
6. PostgreSQL 元数据写入
7. Milvus 向量写入与检索
8. 基于检索结果的大模型问答
9. 单 Agent 工具调用知识检索能力
10. Streamlit 演示页面

### 3.2 第一版暂不实现

1. 多 Agent
2. 父子切块
3. MinerU 多模态解析
4. Rerank 模型
5. 外部订单系统对接
6. 部门权限或用户权限系统
7. 面向生产的复杂部署编排

## 4. 业务场景定位

本项目虽然技术能力具有一定通用性，但产品叙事固定为电商客服场景。

知识库文档建议围绕以下类型组织：

1. 商品资料
2. 售后政策
3. 物流规则
4. 活动规则

系统在回答时的默认角色是电商客服助手，不强调泛行业通用平台，而强调在电商客服问答中的落地应用价值。

## 5. 技术选型与理由

### 5.1 FastAPI

理由：

1. 适合构建清晰的 API 服务
2. 与 Pydantic、SQLAlchemy 等生态兼容良好
3. 自带 Swagger 文档，适合开发调试和演示

### 5.2 Streamlit

理由：

1. 可以快速做出演示页面
2. 降低前端开发复杂度
3. 适合作为第一版验证知识库管理和聊天流程的 UI 壳

### 5.3 PostgreSQL

理由：

1. 适合存储结构化元数据
2. 可用于管理文档、切片、会话、消息等业务信息
3. 相比 SQLite 更有工程化味道，也更符合本项目学习目标

### 5.4 Milvus

理由：

1. 作为专用向量数据库，更接近真实 AI 应用系统常见架构
2. 可以帮助理解向量存储、索引和检索流程
3. 便于后续扩展更大规模知识库

## 6. 系统整体流程

### 6.1 离线知识入库流程

1. 用户通过前端上传文件
2. FastAPI 接收文件并保存到本地 `uploads/`
3. 计算文件 MD5
4. PostgreSQL 检查文档是否已存在
5. 若不存在，则解析正文内容
6. 对正文进行递归切块
7. 对 chunk 批量生成 embedding
8. chunk 元数据写入 PostgreSQL
9. 向量写入 Milvus
10. 更新文档处理状态为成功或失败

### 6.2 在线问答流程

1. 用户输入问题
2. 问题文本生成 embedding
3. Milvus 检索最相近的 chunk 向量
4. 根据 chunk_id 到 PostgreSQL 查询 chunk 内容和来源信息
5. 组织上下文 prompt
6. 调用大模型生成最终答案
7. 将本轮问答写入 PostgreSQL
8. 返回前端展示

### 6.3 单 Agent 流程

1. 用户输入问题
2. Agent 判断是否需要调用知识检索工具
3. 工具内部执行向量检索
4. 工具返回检索结果摘要与引用来源
5. 模型基于工具结果输出最终回答

## 7. 数据分层原则

### 7.1 PostgreSQL 负责

1. 文档信息
2. 文件 MD5
3. 文档类型与状态
4. chunk 文本内容
5. chunk 与 document 关联关系
6. 会话信息
7. 消息记录

### 7.2 Milvus 负责

1. chunk_id 对应的 embedding 向量
2. 向量相似度检索

### 7.3 两者关联方式

Milvus 中保存的主键与 PostgreSQL 中的 `chunk_id` 一一对应。检索时先由 Milvus 返回 `chunk_id`，再由 PostgreSQL 读取正文与来源元数据。

## 8. 第一版模块划分原则

1. `api` 层只做请求接收和响应返回
2. `services` 层负责业务逻辑
3. `repositories` 层负责数据库访问
4. `rag` 层负责检索链路和向量相关逻辑
5. `agent` 层负责工具封装与单 Agent 逻辑
6. `schemas` 层负责输入输出模型
7. `core` 层负责配置、数据库连接和通用基础设施

## 9. 第一版风险点

1. Milvus 本地部署和连接可能存在环境坑
2. PostgreSQL 连接与表结构初始化可能出错
3. PDF 与 DOCX 文档解析效果不稳定
4. 中文切块效果可能影响检索质量
5. Agent 工具调用逻辑在第一版容易引入额外调试成本

## 10. 第一版开发策略

优先验证主链路，不优先堆复杂能力。

推荐开发顺序：

1. 建立项目目录和文档
2. 准备配置项和环境变量
3. 连接 PostgreSQL 和 Milvus
4. 完成文档上传与元数据入库
5. 完成文档解析、切块、embedding、向量入库
6. 完成基础 RAG 检索问答
7. 完成会话与消息存储
8. 完成单 Agent 工具封装
9. 完成 Streamlit 页面

## 11. 项目完成标准

当满足以下条件时，第一版可视为完成：

1. 文档可以上传并成功入库
2. 重复文件可以被识别
3. Milvus 中能查到对应向量
4. 提问后系统可以返回基于知识库的答案
5. 会话记录能写入 PostgreSQL
6. 单 Agent 能调用检索工具
7. 页面能够完成上传和聊天演示

## 12. 第一版阶段结论

当前项目已进入第一版收官阶段。

第一版的重点不再是继续横向叠加能力，而是确认以下核心闭环已经成立：

1. 离线知识入库链路成立
2. 在线问答链路成立
3. 单 Agent 工具调用链路成立
4. PostgreSQL 记忆持久化链路成立
5. 前后端演示链路成立

第一版后续主要工作是复盘实现细节、稳定演示流程、为第二版优化预留清晰接口。

## 13. 第二版目标

第二版聚焦 RAG 效果优化，而不是继续横向堆功能。

第二版核心目标：

1. 让检索结果更相关
2. 让上下文拼接更合理
3. 让模型最终回答更稳定
4. 让整个系统更接近真实业务中的知识问答质量要求

第二版主要建设内容：

1. 数据清洗
2. 切块策略优化
3. 父子分块方案
4. Top-K 与上下文组织方式调优
5. 引入 rerank 重排序
6. 视需要补充 chunk 元数据结构

## 14. 第三版目标

第三版在第二版 RAG 主链路优化完成之后展开，目标是让项目从“知识问答系统”升级为“业务型单 Agent 客服系统”。

第三版主要建设内容：

1. 保留本地知识库检索工具
2. 增加订单查询工具
3. 增加物流、售后、活动规则等业务工具
4. 让 Agent 按问题类型自主选择工具
5. 形成更完整的电商客服场景能力演示
