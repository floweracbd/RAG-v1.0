# 面向电商客服场景的单 Agent RAG 知识服务系统

## 项目简介

这是一个面向电商客服场景的单 Agent RAG 知识服务系统，目标是构建一套可上传商品资料、售后政策、物流规则、活动说明等文档，并基于知识检索完成智能问答的后端应用。

项目第一版聚焦以下核心链路：

1. 文档上传与本地保存
2. 文档解析与 MD5 去重
3. 文本切块与向量化
4. PostgreSQL 存储元数据与会话记录
5. Milvus 存储向量并执行相似度检索
6. 基于检索结果完成 RAG 问答
7. 将知识检索能力封装为单 Agent 可调用工具
8. 使用 Streamlit 提供简易演示页面

## 项目定位

这是一个偏工程化的 AI 应用项目，不是只做文件上传和聊天的演示壳。

对外描述建议使用下面这句话：

> 本项目是一个面向电商客服场景的单 Agent RAG 知识服务系统，支持商品资料和售后规则文档上传、解析、切块、向量化入库，并基于 Milvus 检索和大模型生成完成智能问答，同时使用 PostgreSQL 管理文档元数据、知识片段和会话记录。

## 第一版目标

第一版只追求一件事：

把知识库上传、入库、检索、问答、单 Agent 工具调用这条链路完整跑通，并形成一个可以用于演示和面试讲解的小型项目。

## 第一版功能范围

### 已确定实现

1. 支持上传 `pdf`、`txt`、`md`、`docx` 文档
2. 计算文件 MD5，防止重复入库
3. 解析文档文本内容
4. 使用普通递归切块完成文本分片
5. 生成 embedding 并写入 Milvus
6. 将文档、切片、会话、消息等信息写入 PostgreSQL
7. 提供基于 RAG 的问答能力
8. 提供单 Agent 调用知识检索工具的能力
9. 使用 Streamlit 做知识库管理和聊天演示页面
10. 使用 FastAPI 作为核心后端服务

### 第一版暂不实现

1. 多 Agent
2. MinerU 多模态解析
3. 父子切块
4. 重排序模型
5. 真实订单系统对接
6. 权限系统
7. 平台级多租户支持

## 技术栈

- 后端框架：FastAPI
- 演示前端：Streamlit
- 关系型数据库：PostgreSQL
- 向量数据库：Milvus
- 大模型调用：OpenAI 兼容接口
- Embedding：兼容 embedding API
- 文档解析：pypdf / python-docx / markdown / txt 原生读取

## 系统核心流程

### 离线知识入库流程

1. 用户上传文档
2. 后端保存原始文件
3. 计算 MD5 并检查是否重复
4. 解析文档正文
5. 文本切块
6. 生成 chunk embedding
7. PostgreSQL 写入文档和 chunk 元数据
8. Milvus 写入向量
9. 更新文档处理状态

### 在线问答流程

1. 用户输入问题
2. 系统生成查询向量
3. Milvus 检索 Top-K chunk
4. PostgreSQL 根据 chunk_id 读取内容与来源
5. 拼接上下文
6. 调用大模型生成答案
7. 保存会话与消息记录
8. 返回结果给前端

### 单 Agent 调用流程

1. 用户输入问题
2. Agent 判断是否需要调用知识检索工具
3. 若调用，则执行 RAG 检索
4. 将检索结果回填给模型
5. 模型生成最终答复

## 当前阶段说明

当前项目可以视为第一版收官阶段。

第一版当前已经完成的核心目标是：

1. 跑通知识库上传、入库、检索、问答主链路
2. 将知识库检索能力封装为单 Agent 可调用工具
3. 使用 PostgreSQL 承担会话记忆持久化
4. 使用 FastAPI + Streamlit 完成前后端演示链路

这一阶段后续不再优先扩展复杂功能，而是先复盘现有实现、理解代码细节，并为第二版优化做准备。

## 第二版规划

第二版重点不再是继续加功能，而是集中优化 RAG 效果，让系统从“能跑”进一步提升到“回答更准、更稳”。

第二版重点方向：

1. 数据清洗：清理脏文本、重复内容、无效换行和低价值片段
2. 切分优化：重新评估 chunk size、chunk overlap 与不同文档类型的切分策略
3. 父子分块：引入 parent-child chunk 方案，兼顾召回精度与上下文完整性
4. 检索调优：优化 Top-K、召回策略与上下文拼接方式
5. 重排序：在向量召回之后加入 rerank，提升送入模型的上下文质量
6. 引用增强：逐步补充来源展示与命中文档可解释性
7. 结构补全：在第二版后段视优化结果决定是否补充 chunk 元数据入 PostgreSQL

第二版的目标是形成一条可以清楚讲述的 RAG 优化路径，而不是单纯堆新模块。

## 第三版规划

第三版重点从“知识问答”走向“业务型 Agent”，让系统更贴近真实电商客服场景。

第三版重点方向：

1. 保留知识库检索工具作为基础能力
2. 新增订单查询工具，例如查询订单状态、物流状态、售后状态
3. 逐步补充活动规则、优惠规则、售后政策等业务查询工具
4. 让单 Agent 根据用户问题自主判断调用知识工具还是业务工具
5. 将项目从单纯 RAG Demo 提升为面向电商客服场景的业务型智能助手
